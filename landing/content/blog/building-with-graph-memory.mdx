Agent memory is fundamentally a graph problem. Entities are nodes, relationships are edges, and facts are properties on those nodes and edges.

## Why Graphs

Traditional vector databases store isolated embeddings — they're great for similarity search but terrible for reasoning about relationships. A graph database gives you:

1. **Entity resolution** — "Is this the same user mentioned in session A and session B?"
2. **Relationship traversal** — "Which documents did this agent reference last week?"
3. **Temporal reasoning** — "What was the state of this conversation before the error?"

## OpenZync's Graph Abstraction

OpenZync abstracts over multiple graph backends with a unified query interface:

```python
from packages.graph_backend import PostgresGraphBackend

backend = PostgresGraphBackend(db_session)
entity = await backend.create_entity(org_id, project_id, name="Alice", entity_type="user")
await backend.create_relationship(org_id, project_id, source_id=entity["id"], target_id=doc_id, relationship_type="accessed")
results = await backend.search_entities(org_id, project_id, query="Alice")
```

## Production Considerations

When running graph memory in production, consider:

- **Connection pooling** — Each backend has different pooling semantics
- **Query optimization** — Use parameterized queries and indexed properties
- **Backup strategy** — Graph DBs need different backup approaches than relational DBs
