Observability is often an afterthought in agent systems — but it shouldn't be. When your agent is making wrong decisions and you can't figure out why, you need more than just logs.

## The Three Pillars

### 1. Structured Logging

Every agent action in OpenZync produces a structured log entry:

```python
logger.info("agent.tool_executed", extra={
    "agent_id": "cust-svc-01",
    "tool_name": "search_knowledge_base",
    "duration_ms": 342,
    "token_usage": {"prompt": 150, "completion": 42},
})
```

This means you can filter, aggregate, and alert on specific patterns without parsing text.

### 2. Distributed Tracing

Agent workflows often span multiple services — LLM calls, database queries, tool executions. OpenZync propagates a `request_id` across every boundary:

- API → Worker → LLM → Database
- Each hop records timing and status
- A single trace shows you exactly where time was spent

### 3. Business Metrics

Beyond technical metrics, you need to know:

- How many sessions are created per minute?
- What's the average tool execution time?
- Which tools fail most often?
- What's the cost per agent conversation?

OpenZync exports all of these as Prometheus metrics by default.
